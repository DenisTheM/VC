1. Uebersicht

Die Entwicklungsagenten sind Custom Slash Commands fuer Claude Code, die haeufige Entwicklungsaufgaben automatisieren. Sie sind unter .claude/commands/ im Projektverzeichnis definiert und koennen direkt in einer Claude Code Session aufgerufen werden.

Command Zweck Dauer
/test Build pruefen + Integration Tests ausfuehren 1-3 Min.
/deploy Frontend + Edge Functions deployen 1-2 Min.
/audit Security-Audit der gesamten Codebase 2-5 Min.
/quality Code-Qualitaet und Best Practices pruefen 2-4 Min.

Wichtig: Alle Commands werden innerhalb der Claude Code CLI ausgefuehrt. Sie sind keine eigenstaendigen Skripte, sondern Anweisungen die Claude Code interpretiert und als strukturierte Aufgaben abarbeitet.

2. Voraussetzungen

Installierte Tools

  • Claude Code CLI — claude muss installiert und konfiguriert sein
  • Node.js — fuer npm run build und Test-Ausfuehrung
  • Supabase CLI — fuer Edge Function Deployments (npx supabase)
  • Vercel CLI — fuer Frontend Deployments (npx vercel)

Konfigurationsdateien

Datei Benoetigt fuer Inhalt
.env.test /test (Integration Tests) TEST_EMAIL, TEST_PASSWORD
~/.supabase/access-token /deploy (Edge Functions) Supabase Access Token
~/.vercel/ /deploy (Frontend) Vercel Login Session

Achtung: Die .env.test Datei enthaelt echte Zugangsdaten und darf niemals committed werden. Sie ist in .gitignore eingetragen.

3. /test — Test-Runner Agent

/test

Fuehrt einen vollstaendigen Build-Check und die Integration Tests aus. Analysiert die Ergebnisse und meldet Probleme.

Was passiert

  1. Build-Pruefung: npm run build wird ausgefuehrt. Bei TypeScript-Fehlern werden diese aufgelistet und Fixes vorgeschlagen.
  2. Integration Tests: node test-full.mjs startet die Test-Suite mit 18 Modulen und 52+ Tests gegen die live Supabase-Instanz.
  3. Analyse: Claude wertet die Ausgabe aus — PASS/FAIL/WARN Zaehlung, kritische Fehler hervorheben.
  4. Zusammenfassung: Kurzbericht mit Status und empfohlenen naechsten Schritten.

Aufruf

/test

Keine Parameter. Der Agent entscheidet automatisch, ob Integration Tests moeglich sind (abhaengig von .env.test).

Getestete Bereiche

ModulTests
AuthentifizierungLogin, Profil, JWT-Ablauf
OrganisationenLaden, Datenqualitaet (Land, SRO, Kontakt)
Company ProfilesVollstaendigkeit, Pflichtfelder
DokumenteCRUD, Content-Qualitaet, Versionshistorie
DokumentgenerierungClaude API, Custom Chapters, Formatierung
Status-Workflowdraft → review → current → outdated
Regulatory AlertsLaden, betroffene Clients, Aktionen
NotificationsSystem-Benachrichtigungen
MessagingAdmin-Nachrichten senden
Kunden (KYC)Endkunden, Reviews, Risiko-Levels
ZefixHandelsregister-Integration
Security (RLS)Row Level Security, Zugriffsrechte
Edge Function ErrorsFehlende Parameter, ungueltige Requests
DatenintegritaetVerwaiste Datensaetze, Konsistenz

Beispiel-Ausgabe

Build: OK (1.12s, keine Fehler)

Integration Tests:
  PASS: 48  |  FAIL: 2  |  WARN: 4

Kritische Probleme:
  1. Document Generation timeout (>120s) — Claude API ueberlastet
  2. Notification System inaktiv — keine Benachrichtigungen generiert

Empfehlung: Generation retry-logic pruefen, Notification-Trigger testen

4. /deploy — Deploy Agent

/deploy [ziel]

Baut das Frontend und deployed es auf Vercel und/oder die Edge Functions auf Supabase. Erkennt automatisch welche Functions sich geaendert haben.

Parameter

AufrufVerhalten
/deployAlles deployen (Frontend + geaenderte Edge Functions)
/deploy frontendNur Frontend auf Vercel
/deploy functionsNur geaenderte Edge Functions
/deploy generate-documentNur eine bestimmte Edge Function

Was passiert

  1. Pre-Flight Check: Uncommitted Changes? Build erfolgreich?
  2. Frontend: npm run build → npx vercel --prod
  3. Edge Functions: Git-Diff pruefen → nur geaenderte Functions deployen via npx supabase functions deploy <name> --no-verify-jwt
  4. Post-Deploy: Zusammenfassung mit URLs und Status

Verfuegbare Edge Functions

FunctionBeschreibung
generate-documentCompliance-Dokument via Claude API generieren
send-client-messageAdmin-Nachricht an Organisation senden
zefix-lookupSchweizer Handelsregister abfragen
invite-memberNeues Organisations-Mitglied einladen
notify-approvalFreigabe-Benachrichtigung senden
notify-alertAlert-Benachrichtigung an betroffene Clients
notify-help-requestHilfe-Anfrage Benachrichtigung
check-action-remindersErinnerungen fuer offene Massnahmen
check-customer-reviewsFaellige Kunden-Reviews pruefen

Tipp: Nach groesseren Aenderungen empfiehlt sich der Ablauf: /test → Commit → /deploy. So stellst du sicher, dass nur getesteter Code deployed wird.

5. /audit — Security Audit Agent

/audit

Fuehrt einen umfassenden Security-Audit der gesamten Codebase durch. Prueft CORS, Authentifizierung, Input-Validierung, Secrets und Security Headers.

Pruefbereiche

CORS-Konfiguration

Sucht in allen Edge Functions nach Access-Control-Allow-Origin. Wildcard (*) wird als Risiko gemeldet — sollte auf eigene Domains beschraenkt werden.

Authentifizierung & Autorisierung

  • Auth-Header Pruefung in allen Edge Functions
  • Rollen-Pruefung (admin/client) vorhanden?
  • Passwort-Policy in supabase/config.toml

Input-Validierung & Injection

  • dangerouslySetInnerHTML mit DOMPurify?
  • SQL Injection Risiken (String-Interpolation in Queries)
  • HTML-Escaping in Email-Templates

Secrets & Konfiguration

  • Keine .env-Dateien im Git
  • Keine hardcodierten API Keys
  • .gitignore vollstaendig

Content Security Policy

  • Security Headers in vercel.json
  • CSP, X-Frame-Options, HSTS

Ergebnis-Kategorien

SeverityBedeutungAktion
KRITISCHSicherheitslueckeSofort beheben
HOCHKonfigurationsproblemInnerhalb 1 Woche
MITTELBest-Practice VerbesserungEinplanen
INFOHinweisOptional

6. /quality — Code Quality Agent

/quality [bereich]

Prueft die Codequalitaet: TypeScript-Typen, Performance-Probleme, toter Code und Abhaengigkeiten.

Parameter

AufrufVerhalten
/qualityVollstaendiger Quality-Check
/quality typesNur TypeScript-Typen (any, @ts-ignore)
/quality performanceNur Performance (N+1 Queries, Bundle Size)
/quality dead-codeNur unbenutzter/toter Code

Pruefbereiche

TypeScript-Qualitaet

  • Build-Fehler via npm run build
  • any Types aufspueren und typisierte Alternativen vorschlagen
  • @ts-ignore / @ts-expect-error Kommentare pruefen

Performance

  • N+1 Queries: Schleifen mit DB-Queries in Edge Functions
  • React Re-Renders: Fehlende useMemo/useCallback
  • Bundle Size: Grosse Imports die ersetzt werden koennten

Toter Code

  • Exportierte Functions/Components ohne Import
  • Auskommentierte Code-Bloecke
  • Legacy Edge Functions (Nochda/Stillalive) die nicht mehr referenziert werden

Abhaengigkeiten

  • npm audit fuer bekannte Schwachstellen
  • Veraltete Packages mit Major-Version-Rueckstand

8. Troubleshooting

"/test" — Integration Tests werden uebersprungen

Die Datei .env.test fehlt oder enthaelt keine gueltigen Credentials. Erstelle sie im Projektroot:

TEST_EMAIL=dein@email.com
TEST_PASSWORD=deinPasswort

"/deploy" — Vercel-Fehler "not linked"

Einmalig npx vercel link ausfuehren und das Projekt verknuepfen.

"/deploy" — Supabase "not logged in"

Einmalig npx supabase login ausfuehren.

"/audit" oder "/quality" — Dauert sehr lange

Diese Commands durchsuchen die gesamte Codebase. Bei grossen Projekten kann das 3-5 Minuten dauern. Das ist normal.

Commands werden nicht erkannt

Stelle sicher, dass du dich im richtigen Projektverzeichnis befindest:

cd ~/Projekte/virtue-compliance
claude

Die .claude/commands/ Dateien muessen im Projektroot liegen.
